# Тема 5: Архитектура

113. Клиент-серверная архитектура — клиент обращается к серверу с запросами, сервер обрабатывает данные. Данные хранятся в БД на стороне сервера.

114. Виды архитектуры: монолитная (единая кодовая база), SOA (крупные функциональные службы), микросервисная (множество мелких независимых сервисов).

115. SOA vs микросервисы: SOA — крупные сервисы, общая БД, взаимодействие через ESB. Микросервисы — узкие задачи, своя БД у каждого, разные языки программирования.

116. Декомпозиция на микросервисы: по бизнес-возможностям (профиль, корзина, оплата, доставка) или через DDD (домены и субдомены).

117. Независимость микросервисов: сбой одного не ломает другие, каждая команда отвечает за свой сервис, выбирает стек и управляет своей БД.

118. Монолит vs микросервисы: монолит — просто на старте, трудно масштабировать, одна ошибка может остановить всё; лучше для маленьких проектов. Микросервисы — независимые команды, разные технологии, точечное масштабирование; минус — сложность инфраструктуры.

119. Хореография — децентрализованный подход, Event-Driven Architecture. Сервисы обмениваются событиями через брокеры (Kafka, RabbitMQ), реагируют самостоятельно.

120. Хореография плюсы: слабая связанность, высокая масштабируемость. Минусы: сложно отслеживать сквозные транзакции и причины сбоев.

121-122. Оркестрация — центральный контроллер управляет вызовами шаг за шагом. Плюсы: прозрачность, легко отслеживать статус и делать откаты. Минусы: контроллер — бутылочное горлышко.

123. AJAX — отправка/получение данных от сервера в фоне. Обновление частей страницы без полной перезагрузки.

124. CAP-теорема: невозможно одновременно обеспечить Консистентность + Доступность + Устойчивость к разделению. Выбирают два из трёх.

125. API Gateway — единая точка входа для клиентских запросов, маршрутизирует к нужным микросервисам.

126. Балансировщик нагрузки — распределяет трафик между серверами для предотвращения перегрузки, обеспечивая доступность и быстродействие.

127. Event Sourcing — состояние хранится как хронологический журнал всех событий (не единая запись). Позволяет восстановить историческое состояние в любой момент.
