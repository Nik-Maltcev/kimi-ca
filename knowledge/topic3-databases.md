# Тема 3: Базы данных

31. БД — упорядоченный набор данных для удобного доступа, поиска, обновления и управления информацией.

32. СУБД — программная система для создания, управления и обеспечения доступа к базам данных.

33. Типы БД: реляционные (таблицы со связями) и нереляционные (NoSQL): ключ-значение (Redis), документо-ориентированные (MongoDB, JSON-документы в коллекциях), графовые (узлы и связи).

34. Реляционные БД называются так от слова «relation» (связь/отношение) — в основе лежат математические отношения наборов данных, информация хранится в связанных таблицах.

35. Задачи нереляционных СУБД: ключ-значение — быстрое кэширование; документо-ориентированные — разнородные данные без жесткой структуры; графовые — поиск сложных взаимосвязей (соцсети, мошенничество, маршруты).

36. Реляционные vs нереляционные: реляционные — жесткие таблицы со связями через ключи. Нереляционные — гибкая структура (графы, JSON, ключ-значение), легко менять формат на лету.

37. Нормализация БД — процесс организации данных для снижения избыточности. Разделение больших таблиц на мелкие по «нормальным формам». На практике обычно до 3НФ.

38. Нормальные формы: 1НФ — нет дублей строк, данные атомарные. 2НФ — есть первичный ключ, все данные зависят только от него. 3НФ — нет транзитивных зависимостей (неключевые столбцы не зависят от других неключевых).

39. Денормализация — осознанное добавление избыточных данных для оптимизации скорости чтения в высоконагруженных системах. Снижает количество сложных JOIN.

40. Виды связей: один к одному (1:1), один ко многим (1:M), многие ко многим (M:N).

41. Связь многие ко многим реализуется через промежуточную таблицу-посредник с ключами обеих сущностей.

42. Первичный ключ (Primary Key) — уникальный идентификатор записи в таблице. Свойства: уникальность и обязательность (не NULL), неизменяемость, автоматическая генерация и индексирование.

43. Транзакция — набор логически связанных операций с БД, выполняемых как единое целое. Успех — Commit, ошибка — Rollback.

44. ACID: Атомарность (полностью или никак), Консистентность (корректный переход между состояниями), Изолированность (параллельные транзакции не влияют друг на друга), Надежность (зафиксированные изменения не потеряются при сбоях).

45. Проектирование БД: сбор требований → выделение сущностей и атрибутов → определение связей и ER-диаграмма → нормализация до 3НФ → документация.

46. Оптимизация БД: нормализация/денормализация, правильные типы данных, индексы, оптимизация SQL (JOIN вместо подзапросов, без SELECT *), кэширование (Redis), партиционирование и шардирование.

47. Партиционирование — логическое разбиение большой таблицы на части для ускорения поиска. Типы: Range (по диапазонам), Hash (равномерное распределение), List (по группам), композитное.

48. Реплицирование — копирование данных между серверами (master для записи, slave для чтения). При сбое master — slave занимает его место.

49. Шардирование — горизонтальное масштабирование, части таблицы хранятся на независимых серверах. Масштабирует и чтение, и запись. Сложнее в администрировании.

50. Реплицирование vs шардирование: репликация — одинаковые копии данных (масштабирует чтение, отказоустойчивость). Шардирование — уникальные куски данных на разных машинах (масштабирует и чтение, и запись, сложнее).

51. Маппинг данных — сопоставление и преобразование полей из одного формата в другой при интеграциях (CSV→БД, JSON API→таблицы).

52. Уровни абстракции БД: концептуальный (бизнес-модель, ключевые сущности), логический (атрибуты, типы данных, ограничения, связи), физический (конкретная СУБД, таблицы, столбцы, ключи, индексы).

53. Индексы — ускорение поиска и сортировки (как оглавление в книге). Минус: замедляют INSERT/DELETE из-за обновления индексов.

54. Грязное чтение — транзакция читает данные, измененные другой незавершенной транзакцией. Решение: уровень изоляции READ COMMITTED, блокировки, версионирование данных.

55. Команды SQL: DDL (CREATE, ALTER, DROP — управление структурой) и DML (SELECT, INSERT, UPDATE, DELETE — управление данными).

56. JOIN: INNER JOIN (только совпадения), LEFT JOIN (все из левой + совпадения), RIGHT JOIN (все из правой + совпадения), FULL JOIN (все из обеих таблиц).

57. TRUNCATE vs DELETE: TRUNCATE удаляет все строки мгновенно и необратимо, не проверяет целостность, не пишет в журнал. DELETE позволяет удалять по условию и откатывать.

58. HAVING — фильтрация сгруппированных данных по агрегатным функциям (SUM, COUNT, AVG). WHERE не может работать с агрегатами.

59. HAVING vs WHERE: WHERE фильтрует строки до GROUP BY. HAVING фильтрует группы после GROUP BY.

60. Выбрать данные из первой таблицы, которых нет во второй: LEFT JOIN + IS NULL, NOT EXISTS, NOT IN, EXCEPT/MINUS.

61. UNION vs UNION ALL: UNION удаляет дубликаты (медленнее). UNION ALL оставляет все как есть (быстрее).
