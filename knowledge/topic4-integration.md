# Тема 4: Интеграция и API

62. Синхронное vs асинхронное: синхронное — блокирующее (клиент ждет ответа). Асинхронное — отправитель не ждет, продолжает работу.

63. В синхронном запросе ответ ожидается всегда. В асинхронном — ответ придет позже или может не прийти.

64. Пример синхронного: приложение магазина отправляет запрос в БД и блокируется, ожидая список книг.

65. Очереди в асинхронных запросах: повышают отказоустойчивость, сообщения ждут в очереди если получатель недоступен, отправитель не блокируется.

66. Вызов хранимой процедуры в БД — синхронный метод (приложение блокируется до получения результата).

67. API — программный интерфейс и правила, по которым приложения общаются друг с другом. Для стандартизации обмена данными, безопасности и реализации бизнес-процессов между системами.

68. REST — архитектурный стиль для веб-приложений поверх HTTP(S). Принципы: ресурсы с URI, унифицированный интерфейс (CRUD через GET/POST/PUT/DELETE), Stateless, кэширование, слоистая архитектура.

69. Restful-приложение — продукт, спроектированный в строгом соответствии с принципами REST.

70. Stateless — сервер не запоминает предыдущие запросы/сессии клиента. Каждый запрос независим и содержит всю информацию. Упрощает архитектуру, повышает масштабируемость.

71. Кэширование — сохранение результатов для повторного использования. Снижает нагрузку на сервер, ускоряет отклик.

72. Кэширование реализуется на стороне клиента (браузер), сервера или промежуточного API.

73. HTTP-методы REST: GET (чтение), POST (создание), PUT (полное обновление/создание), PATCH (частичное обновление), DELETE (удаление). Вспомогательные: OPTIONS, HEAD, TRACE.

74. Передача параметров: Query Parameters (?id=123), Path Parameters (/resource/123), Request Body (тело запроса), Headers (заголовки — метаданные и токены).

75. URL в REST: протокол + домен, версия API (/v1/), путь к ресурсу (/books/123), параметры запроса (?page=2).

76. HEADER ответа: Content-Type, Content-Length, дата, сервер, Cache-Control, ETag, Set-Cookie, Allow.

77. 200 OK — успешная обработка запроса. 201 Created — успех + создан новый ресурс (обычно POST).

78. POST vs GET: GET — чтение, кэшируется, идемпотентен, ограничения URL. POST — создание, данные в теле, не кэшируется, не идемпотентен.

79. PUT vs PATCH: PUT — полная замена ресурса (все атрибуты). PATCH — частичное обновление (только измененные поля).

80. POST vs PUT: POST — создание, ID генерирует сервер (/orders/). PUT — создание/обновление, ID известен клиенту (/orders/123).

81. Идемпотентность — многократное выполнение = тот же результат. Методы GET, PUT, DELETE идемпотентны. Важно для надежности при обрывах сети.

82. DELETE идемпотентен — повторное удаление не вызывает ошибок, просто подтверждает отсутствие ресурса.

83. POST можно сделать идемпотентным через уникальные ключи или токены идемпотентности в заголовках.

84. POST для получения данных — допустимо когда параметры слишком велики для URL или содержат конфиденциальные данные.

85. Обратная совместимость — через версионирование API. Новая мажорная версия только при ломающих изменениях (удаление полей, смена структуры ответа).

86. JSON Schema — стандарт для валидации JSON: обязательные поля, типы данных, лимиты длины, уникальность массивов.

87. HTML vs XML: HTML — фиксированные теги для отображения в браузере. XML — свои теги для передачи структурированных данных между машинами.

88. XML содержит: Объявления (версия, кодировка), Элементы (родительские/дочерние теги), Атрибуты (параметры тегов), Комментарии.

89. XSD — язык описания структуры XML-документа: теги, последовательность, обязательность, типы данных.

90. Пространство имен (Namespace) в XML — механизм разрешения конфликтов одинаковых названий тегов в разных контекстах.

91. Работа с XML: текстовые редакторы (Notepad++), IDE (IntelliJ, VS), визуальные редакторы (XMLSpy, Oxygen), библиотеки языков (Java, Python, C#).

92. SOAP — протокол обмена сообщениями в формате XML. Встроенная безопасность (WS-Security). Стандарт для банков, телеком, госуслуг.

93. SOAP-сообщение: Envelope (оболочка) → Body (бизнес-данные) + опционально Fault (информация об ошибках).

94. WSDL — XML-контракт SOAP-сервиса: методы, входные/выходные форматы, порты, протоколы.

95. SOAP vs REST: REST — гибкий стиль, любые форматы (JSON/XML), кэширование, легкий. SOAP — строгий протокол, только XML, WSDL-контракт, высокая безопасность (SSL + WS-Security), не кэшируется.

96. Очереди сообщений: сглаживание пиковых нагрузок, отложенная обработка без блокировки, гарантия доставки при недоступности получателя.

97. Kafka vs RabbitMQ: RabbitMQ — Push-модель, сложная маршрутизация через обменники, удаляет сообщение после ACK. Kafka — Pull-модель, хранит сообщения в топиках на диске, миллионы msg/sec, можно перечитывать.

98. Гарантия доставки: RabbitMQ — ACK + зеркалирование очередей. Kafka — репликация + хранение на диске.

99. Перечитать сообщения в Kafka: управление оффсетом — запомнить позиции, запросить данные с старых оффсетов, обработать, обновить оффсет.

100. Очередь vs топик: очередь — один-к-одному, сообщение удаляется после извлечения (FIFO). Топик — подписка, копия всем подписчикам, данные хранятся.

101. Корпоративная шина (ESB) — узел, объединяющий системы. Маршрутизация, трансформация форматов (XML↔JSON), контроль безопасности.

102. Шина vs очередь: очередь — один источник → один получатель. Шина — публикация → все подписанные системы получают копию.

103. ESB vs ETL: ESB — мгновенный обмен сообщениями в реальном времени. ETL — пакетная выгрузка/трансформация/загрузка больших массивов данных по расписанию.

104. Новые обязательные поля в веб-сервисе на шине: обновить контракты, задокументировать поля, доработать зависимую логику, сквозное тестирование.

105. GraphQL — язык запросов для API (Facebook). Данные как граф, единая точка входа. Query (получение), Mutation (изменение), Subscription (подписка на обновления).

106. GraphQL vs REST: решает Overfetching/Underfetching — клиент запрашивает ровно нужные поля, меньше запросов, меньше трафика.

107. WebSocket — постоянное двустороннее TCP-соединение после Handshake. Сервер сам отправляет данные клиенту в реальном времени. Чаты, биржи, игры.

108. gRPC — высокопроизводительный RPC-фреймворк (Google). Для быстрых внутренних API между микросервисами.

109. Webhook — обратный вызов. Клиент оставляет URL, сервер при событии сам отправляет POST. Мгновенная реакция на события, снижение нагрузки.

110. Webhook плюсы: мгновенность, экономия ресурсов, простота, масштабируемость. Минусы: нет гарантий доставки, уязвимость URL (нужна проверка подписи).

111. Файловый обмен — интеграция через выгрузку/передачу файлов когда нет прямого API. Пример: выгрузка контактов из CRM в сервис рассылок.

112. Файловый обмен плюсы: простота, надежность (файлы на диске). Минусы: медленность, ограничения на объем больших файлов.
