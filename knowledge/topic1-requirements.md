# Тема 1: Требования

1. Какие есть способы сбора требований? Сбор требований можно проводить множеством разных методов в зависимости от ситуации. Основные из них — это интервью (индивидуальные или групповые беседы со стейкхолдерами) и анкетирование (опросы больших групп пользователей, например, через Google-формы). Также аналитик может использовать полевое наблюдение (изучение работы сотрудника прямо на его рабочем месте), мозговые штурмы для генерации идей и фокус-группы для обсуждения проблем интеграции между системами. Дополнительно применяются анализ документов (особенно важно для соблюдения законов, например, ФЗ, или ГОСТов), анализ существующих интерфейсов и анализ систем-конкурентов.

2. Кто такие стейкхолдеры? Стейкхолдер — это любое лицо или группа людей, которые так или иначе заинтересованы в проекте и разрабатываемой системе. К ним относятся владелец продукта (product owner), отвечающий за рыночный успех системы, команда разработки (которая создает систему по артефактам аналитика), а также конечные пользователи, которые будут напрямую работать в программе. Кроме того, стейкхолдерами являются узкопрофильные эксперты предметной области.

3. Какие есть виды требований? Существует три основных уровня требований: бизнес-требования, пользовательские и функциональные. Бизнес-требования исходят от компании и описывают глобальные цели. Пользовательские требования зависят от бизнеса и описывают, что нужно конкретным ролям для работы. Функциональные требования описывают непосредственно функции системы. Отдельной группой стоят нефункциональные требования, которые описывают качественные характеристики (надежность, скорость, цвета кнопок и т.д.).

4. Какая классификация нефункциональных требований? НФТ делятся на четыре большие группы: требования к внешним интерфейсам, бизнес-правила, ограничения и атрибуты качества. Требования к внешним интерфейсам описывают правила взаимодействия с ОС, сторонними API, оборудованием, а также стандарты UI. Бизнес-правила — это законы, постановления и корпоративные стандарты. Ограничения задают жесткие рамки для разработки. Атрибуты качества определяют, насколько хорошо работает система: внутренние (эффективность, масштабируемость) и внешние (доступность, безопасность, удобство).

5. В чем отличия функциональных и нефункциональных требований? Функциональные требования отвечают на вопрос «ЧТО» должна делать система, а нефункциональные — на вопрос «КАК» она это делает.

6. Что такое транзитные (переходные) требования? Переходные требования — это временные условия, необходимые исключительно на этапе миграции системы из текущего состояния (AS-IS) в новое, целевое состояние (TO-BE). К таким требованиям относятся правила миграции старых баз данных, инфраструктурные изменения, проведение UAT и первичное обучение сотрудников.

7. Что такое User story? User story — это короткое описание функционала с точки зрения конечного пользователя по шаблону: «Как [роль], я хочу [действие], чтобы [цель]».

8. Что такое User story map (USM)? USM — это методология визуализации и планирования задач в Agile-разработке. Она наглядно отображает весь путь пользователя, упорядочивая все шаги и задачи.

9. Преимущества User story: описывает потребность понятным для бизнеса языком, экономит время на согласование, помогает команде понимать ценность продукта, удобна для планирования по спринтам.

10. Как понять, что User story написана хорошо? Чеклист INVEST: Independent (независимая), Negotiable (обсуждаемая), Valuable (ценная для бизнеса), Estimable (оцениваемая), Small (компактная для одного спринта), Testable (тестируемая).

11. Что такое Use cases? Use Case детально описывает взаимодействие системы с внешними акторами. Включает заголовок, акторов, триггер, предусловия, постусловия, основной сценарий и альтернативные сценарии.

12. В чем отличие User stories от Use cases? User story описывает высокоуровневую потребность пользователя. Use case описывает функциональные требования — строгую последовательность технических шагов.

13. ГОСТ 19 и ГОСТ 34 — российские стандарты для технической документации. Важны для госзаказов.

14. Разница между ГОСТ 19 и ГОСТ 34: ГОСТ 19 — для ПО, ГОСТ 34 — для комплексных автоматизированных систем (ПО + оборудование).

15. Критерии хороших требований: понятные, полные, однозначные, осуществимые, измеримые, необходимые, гибкие, верифицируемые.

16. Артефакты работы аналитика: BPMN-схемы, User stories, глоссарии, ER-диаграммы, спецификации API, ТЗ, спецификация сервиса, постановки задач.

17. Пользователи артефактов: разработчики, QA, продакты, руководители, техподдержка, конечные пользователи.

18. Шаблон постановки включает: общую часть (контекст, цели, глоссарий), паспорт (название, ссылки, роли, НФТ, критерии приемки), логику для фронтенда и бэкенда, блок для тестирования.

19. Для UI в постановке: макеты, таблица элементов (поля, кнопки, типы данных, обязательность), схемы переходов, маппинг данных, взаимодействие с бэкендом.

20. Описание интеграции: интегрируемые системы, регламент, тип (API, очередь), протокол, безопасность (токены), маппинг форматов, обработка ошибок, примеры запросов/ответов, Sequence-диаграммы.

21. SRS — спецификация требований к системе. Единый документ с ФТ, НФТ, описанием аудитории, зависимостями. Создается на старте проекта.

22. Верификация vs валидация: верификация — проверка правильности написания требований (соответствие стандартам). Валидация — проверка того, что требования несут нужную бизнесу ценность.
