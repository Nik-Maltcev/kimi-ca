# Тема 6: Методологии разработки

128. Этапы разработки: Сбор/анализ требований → Проектирование (Use cases, User stories, архитектура, БД, интерфейсы) → Оценка сроков → Разработка (код) → Тестирование → Внедрение (деплой, обучение) → Поддержка/Сопровождение.

129. Роль аналитика: связующее звено между бизнесом и командой. Сбор требований, проектирование, контроль разработки, консультация программистов, демонстрация функционала заказчикам.

130. Обязательное участие аналитика: сбор требований, проектирование, демонстрация (сдача). Опциональное: разработка и тестирование (роль консультанта).

131. Аналитик + тестировщики: передает документацию, помогает писать тест-кейсы, может сам тестировать API или проверять UI на соответствие макетам.

132. Декомпозиция задач — подход SPIDR: Spikes (прототипы), Paths (альтернативные сценарии), Interfaces (по устройствам: веб, iOS), Data (по типам данных/регионам), Rules (сложные бизнес-правила — отдельно).

133. DoR (Definition of Ready) — задача полностью описана и обеспечена для взятия в спринт. DoD (Definition of Done) — задача написана, протестирована, задокументирована. AC (Acceptance Criteria) — чек-лист условий для приемки функционала.

134. Виды оценки: экспертный (в часах — точное время) и относительный (Story Points/майки — абстрактная сложность).

135. Часы vs Story Points: часы — физическое время. Story Points — сложность + объем + риски неопределенности.

136. Оценка сроков аналитиком: экспертная оценка, декомпозиция (WBS), метод PERT (оптимистичный + пессимистичный + вероятный сценарии).

137. Методологии: Waterfall — последовательные этапы без возврата, хорош для госзаказов с фиксированными требованиями. Scrum — спринты 1-4 недели, регулярная демонстрация. Kanban — непрерывный поток, визуализация, ограничение WIP.

138. Scrum подходит: проекты с неопределенностью (веб/мобильные приложения), частые изменения. Не подходит: жесткая предсказуемость, строгие регламенты, фиксированный бюджет.

139. Мероприятия Scrum: Daily Stand-ups (15 мин), Планирование спринта, Review (демонстрация), Ретроспектива (улучшение процессов), Refinement (уточнение бэклога).

140. Kanban — визуализация работы, минимизация незавершенных задач. 3 правила: визуализация процесса (доски/карточки), ограничение WIP, управление потоком (Cycle Time, устранение узких мест).

141. Kanban vs Scrum: Scrum — фиксированные спринты, роли (Скрам-мастер, PO), изменения нежелательны. Kanban — непрерывный поток, нет ролей/спринтов, задачи добавляются в любой момент.

142. Хороший аналитик: софт-скиллы, умеет общаться с бизнесом, переводит потребности в архитектуру, проактивен, берет ответственность, ищет зоны роста.

143. Джуниор vs сеньор: джуниор — работает под руководством, локальные задачи. Сеньор — самостоятельная архитектура, управление рисками, разрешение конфликтов, наставничество.

144. Плохой аналитик: не понимает IT-технологии, хаотичная документация, не умеет моделировать (UML, BPMN), плохая коммуникация, предлагает костыли.

145. Обязанности системного аналитика: сбор ФТ/НФТ, проектирование БД (SQL, ER), спецификации интеграций (REST, очереди), диаграммы (UML, BPMN), контроль разработки, консультации, инструкции.
